# DSPy: Compiling Declarative Language Model Calls into Self-Improving Pipelines

## 背景
LLM（大規模言語モデル）のアプリケーション構築において、従来は手作業で記述された「プロンプト（指示文や少数の例示を含む長い文字列）」を試行錯誤しながら作成する「プロンプトエンジニアリング」が主流でした。しかし、この文字列ベースのアプローチは非常に脆く（brittle）、異なるモデルやタスク、データの変化に対応しづらいというスケーラビリティの課題がありました。システムが複雑になり複数のLLM呼び出し（推論ループや外部ツールの利用など）が連鎖するようになると、1つの呼び出しの文脈が変わるだけで他のプロンプトも壊れてしまう可能性があります。この課題を解決するため、著者らは言語モデルのプロンプト記述を「テキスト変換グラフ」として抽象化し、プログラマティックに最適化可能な「パラメータを持つモジュール」と「コンパイラ」へと置き換える、新たなプログラミングモデル「DSPy」を提案しました。

## 手法
著者はLLMのパイプラインを構築・最適化するためのフレームワーク「DSPy」を構築しました。DSPyは以下の3つのコア要素で構成されています。

### Signatures（シグネチャ）
LLMへの入出力を定義する「自然言語による型宣言」です。
従来のプロンプトエンジニアリングが、「タスクの背景」「入力の形式」「出力の形式」「文脈の取り扱い」などを長大な文字列テンプレートとして定義していたのに対し、DSPyのシグネチャは `question -> answer` や `context, question -> search_query` といった記述で「何をすべきか（What）」だけを極めてシンプルに宣言します。
内部的には、DSPyのフレームワークがこの短い宣言を解読し、各フィールドのセマンティック（意味論的）な役割に応じた適切なプロンプト命令へ自動的に展開します。（例えば `french_translation` という型であれば自動的にフランス語への翻訳タスクとして解釈されます）。
この抽象化により、開発者はモデルの選定やデータセットに合わせてプロンプトを調整するという脆い（brittle）作業から解放され、再利用可能で汎用的なインターフェースとしてタスクを定義できるようになります。

### Modules（パラメータ化されたモジュール）
PyTorchにおけるディープラーニングの「レイヤー（層）」アーキテクチャにインスパイアされた、オブジェクト指向型の組み立て可能な抽象化モジュールです。
すべての基本となり純粋な入出力を担う `Predict` モジュールを筆頭に、これまで研究で提案されてきた複雑なプロンプト技法をカプセル化した `ChainOfThought` や `ReAct`、`ProgramOfThought`、`MultiChainComparison` など多様な組み込み機能が用意されています。
これらのモジュールの最大の特徴は、静的なプロンプト生成器ではなく、**「学習可能なパラメータ（重み）」を保持する**点にあります。具体的には、使用する「LLMのモデル自体」「前置きの指示文（プレフィックス）」、そして動作の鍵となる「少数の精査された例示（few-shotデモンストレーション）」をパラメータとして内部に抱えています。（※ただし、初代論文である本手法での自動最適化のスコープは主に「few-shot例示」の組み合わせ生成に特化しており、指示文のテキスト自体は更新しません）。
開発者はこれらのモジュールを初期化時に宣言し、Pythonの標準的な制御フロー（`if`文による分岐、`for`ループによる繰り返し、エラーハンドリングなど）を用いた `forward` メソッド内で自由に連結させることで、任意の複雑さを持つ推論パイプライン（テキスト変換グラフ）を柔軟に構築することができます。

### Teleprompters（コンパイラ・最適化ツール）
プログラム（作成したモジュールのパイプライン構成）、最適化したい評価指標（メトリクス）、そしてラベルが不完全でも構わない少数の訓練データを受け取り、前述したモジュールのパラメータを自律的に最適化調整する「オプティマイザ（コンパイラ）」です。
「人がプロンプトを書く」という手法そのものを代替し、手作業のデータアノテーションへの依存を減らしつつ、主に以下の3つの段階（ステージ）を経てパイプラインの品質を最大化します。

* **Stage 1. 実行のトレースと中間ラベルの自己生成 (Bootstrapping)**
  推論プロセス（理由付けや検索クエリなど）の正解データを持たない状態で、初期プログラムに少数の訓練データ（質問と最終回答のみ）を入力し、シミュレーションを実行させます。ある種の「乱れ打ち（Temperatureを上げて多様な推論経路を繰り返し試行する等）」を行う中で、LLMが偶然にもたまたま「正しい最終回答」に辿り着く瞬間が発生すると、その時の**「まぐれ当たりした推論プロセスの軌跡」全体を裏で抽出**します。このようにして、最終的な評価指標（メトリクス）をクリアした成功時の中間生成物だけを、以後の最適化に用いる「質の高い正解データ（擬似ラベル）」としてシステムが自律的に大量蓄積します。
  
* **Stage 2. デモンストレーションの組み合わせ探索と最適化 (Parameter Optimization)**
  Stage 1で引き当てた「偶然出た質の高い推論プロセス」のプールの中から、離散的なハイパーパラメータ最適化の探索アルゴリズムを用いて、最も全体の正答率が安定する「最強の例示の組み合わせ」を特定し、正式なパラメータへと昇格させます。（※本論文のスコープでは指示文の自動最適化は行わず、最適化の対象はこの『例示の選定と組み合わせ』のみに絞られています）。主に以下のアルゴリズムが使用されます。
  * **先着順・ランダム抽出 (`BootstrapFewShot`)**: 指定数（例：4個）の成功軌跡が採取できた時点でそれをそのまま採用する、探索を行わない最も軽量な設定。
  * **ランダムサーチ (`BootstrapFewShotWithRandomSearch`)**: プールから無作為にいくつかの組み合わせのパターンを作り、検証データ上でテストを繰り返して1番スコアが高かった組み合わせを採用する、泥臭いが堅実な設定。
  * **ベイズ最適化 (`BootstrapFewShotWithOptuna`)**: OptunaのTPE (Tree-structured Parzen Estimator) を用い、「どの推論を組み合わせるとスコアが上がるか」の確率モデルを学習しながら、より効率的に最適な組み合わせを狙い撃ちで探索する高度な設定。
  
  結果として、人間が手書きした理由付けよりも「LLM自身が苦労して評価に合格した理由付け」をそのままカンペに使うことで、モデルの性能が極めて高く安定します。（※また、この抽出された例示群を使って、小規模なローカルLLMの重みを直接ファインチューニングする手法へと分岐することも可能です）
  
* **Stage 3. 高次プログラムへの更新と最適化（Higher-Order Optimization）**
  抽出された最適なパラメータ（最良のfew-shot例示セット、またはファインチューニング済みのモデル）を各モジュールに内包させた、新しいコンパイル済みのプログラムを出力します。さらにこの段階で、生成したプログラムを自動で複数並列実行させ、多数決を行って最終推論を決定する「アンサンブル」構造のような、より堅牢なプログラム制御へとメタ的に書き換える最適化（メタプログラミング）も同時に行われます。

## chokosennlovetiの考察

### 先行手法（ProTeGi, OPRO等）と比較したDSPyの新規性と優位性
プロンプト最適化の領域において、ProTeGiやOPROといった先行研究は「テキスト勾配」や「過去の推論軌跡」を用いて**『指示文（プレフィックス）の文字列そのものを自然言語で書き換える』**アプローチをとっていました。しかし、このアプローチには「パイプラインが複雑になると、一箇所の指示文を変更しただけで下流の出力フォーマットが壊れ、システム全体が破綻する」という重大なスケーラビリティの課題（プロンプトの脆さ）がありました。

これに対し、DSPyは以下の点で全く異なるパラダイム（新規性）を提示しています。

1. **文字列操作から「プログラム」への抽象化**
   DSPyはプロンプトエンジニアリングの文法を捨て、タスクを「Signature（自然言語による型宣言）」と「Module（パラメータを持つレイヤー）」というプログラムとして抽象化しました。「How（どう解くか）」の長大な指示文をチューニングするのではなく、「What（入出力の型）」だけを宣言させてテキスト変換グラフを構築できるようにした点が根本的に革新的です。

2. **Bootstrappingによる「中間ラベルの自己生成」の獲得**
   従来のfew-shot学習や最適化では、人間が「中間推論プロセス（Chain of Thoughtの理由付けなど）」を手作業でアノテーションする必要がありました。DSPyは、最終回答させ正解していれば、そこに至ったLLMの「まぐれ当たりの推論の軌跡」から逆算して、システムが自律的に高品質な中間プロセスを大量に抽出（Bootstrapping）できるという突破口を開きました。

3. **「自然言語の書き換え」から「離散的ハイパーパラメータ最適化」への移行**
   DSPyの初期コンパイラは、指示文を捏ね回す代わりに、最適化のスコープを**「抽出したfew-shotデモンストレーションの最適な組み合わせの探索」**に絞り込みました。これをベイズ最適化（TPE / Optuna）のような計算機科学の定石アルゴリズムで解くことで、自然言語による指示の揺らぎ（不確実性）を排除し、どんなに複雑な複数ステップのパイプラインであっても、各モジュールの性能を確実に向上させる「最強のカンペ（例示）」を理論的かつ安定して導き出せるようした点が最大の強みです。

結果として、DSPyは「LLMと逐次改善」の系譜において、単なるプロンプトハックの延長線上にあった先行手法群から完全に分岐し、LLM応用開発を**「スケーラブルなコンパイルベースのソフトウェア工学」**へと昇華させた画期的な手法であると位置づけられます。
