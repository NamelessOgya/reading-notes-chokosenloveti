# CRANE: Reasoning with Constrained LLM Generation

本論文は、厳密な文法制約がLLMの推論能力を理論的に劣化させることを示し、文法拡張（Grammar Augmentation）によって推論と構造制約を両立するデコードアルゴリズム「CRANE」を提案した研究である。

---

## 背景

構造化出力（JSON・コード・形式論理）を保証するためのGCD（文法制約デコード）は、正確な形式の出力を生成できる一方で、タスク精度（推論能力）が低下するという問題が実験的に観察されていた。

しかし、なぜ文法制約が推論能力を劣化させるのかについては、理論的な説明が与えられていなかった。本研究はこの問題に対して初めて厳密な理論的解析を提供し、その解決策としてCRANEを提案した。

---

## 手法

### 1. 理論的解析：文法制約とTC⁰制限

**定理（非公式）**: 十分に制限的な文法制約（例：回答のみを許可する文法）を適用すると、Transformer LLMが実現できる計算クラスは $TC^0$ に制限される。

$TC^0$ クラスは：
- ブール回路の深さが $O(1)$（定数深さ）に制限される計算クラス。
- 乗算・比較など基本算術は表現できるが、**多段階の連鎖推論（Chain-of-Thought）が必要な問題**は解けない。

すなわち、Chain-of-Thought（CoT）がLLMの推論能力を引き出す理由は、中間推論ステップを「記憶外部化」することで回路深さを増やせるためである。しかし厳密な文法制約は中間推論ステップの生成を禁止するため、この恩恵が失われる。

$$\text{文法制約なし}(CoT): \ \text{表現クラス} \supset TC^0$$
$$\text{厳密な文法制約（GCD）}: \ \text{表現クラス} = TC^0$$

### 2. 文法拡張（Grammar Augmentation）

CRANEの中核は制約する文法自体を拡張することにある。

$$G_{\text{augmented}} = G_{\text{target}} \cup G_{\text{reasoning}}$$

- $G_{\text{target}}$: 最終出力に求められる文法（例：有効なJSONスキーマ）
- $G_{\text{reasoning}}$: CoTステップなど中間推論を許可するルール（例：任意のテキストとXMLタグの組み合わせ）

拡張文法では、LLMが「推論フェーズ」（$G_{\text{reasoning}}$）と「出力フェーズ」（$G_{\text{target}}$）を自然に切り替えながら生成できる。

### 3. CRANEデコードアルゴリズム

1. **推論フェーズ（非制約）**: LLMが中間推論ステップ（CoT）を自由に生成する。XGrammarやOutlinesの制約は適用しない。
2. **出力フェーズ（制約付き）**: 特定のフラグトークン（例：`<output>`タグ）を検出した時点から、$G_{\text{target}}$ の文法制約を適用して最終出力を生成する。
3. **反復**: 必要に応じて推論フェーズと出力フェーズを交互に繰り返す。

## 結果

### 推論ベンチマークでの精度改善（主要結果）

| ベンチマーク | 制約なし生成 | GCD（従来手法） | **CRANE** |
| :--- | :---: | :---: | :---: |
| GSM-Symbolic（数学推論） | 高 | 大幅低下（-10〜-15pt） | **ほぼ制約なし生成と同等** |
| FOLIO（一階述語論理推論） | 高 | 大幅低下（-8〜-12pt） | **制約なし生成より高精度** |

- 対GCD比で最大 **10ポイントの精度改善**
- 対制約なし生成では同等以上の精度を維持しながら構造的正確性を100%保証

### 構造的正確性

| 手法 | JSON/コード 構文適合率 |
| :--- | :---: |
| 制約なし生成 | 低い（数%〜数十%のエラー） |
| GCD | 100% |
| **CRANE** | **100%** |

## chokosenlovetiのメモ

### GPU正規表現高速化との関係

CRANEが解決する「最適化の副作用として性能が劣化する」問題構造は、GPU正規表現マッチングの研究と類似している。

- **最適化のトレードオフ**: GPU正規表現研究では、DFA圧縮（メモリ最適化）がウォープ発散を増やして性能を低下させる（C-DFAの問題）。CRANEでは、文法制約（精度最適化）が推論能力を低下させる。いずれも「一つの軸の最適化が別の軸を阻害する」問題。
- **フェーズ分割の設計**: CRANEが推論フェーズと出力フェーズを分離する設計思想は、GPU正規表現マッチングにおけるハイブリッドDFA/NFA（Vasiliadis 2009）の「状態数が少ない正規表現はDFAで、爆発するものはNFAで処理」という役割分担と対応する。
- **LLMの正規表現高速化への示唆**: LLMの推論能力を保持しながら正規表現マッチングを高速化するためには、CRANEのように「推論フェーズは自由に、検証フェーズは厳密に」という段階的な制約適用が有効である可能性がある。
