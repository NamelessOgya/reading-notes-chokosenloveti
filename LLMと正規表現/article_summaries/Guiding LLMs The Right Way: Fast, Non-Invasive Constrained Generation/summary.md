# Guiding LLMs The Right Way: Fast, Non-Invasive Constrained Generation (DOMINO)

本論文は、サブワードトークナイザと正規表現・CFGの境界不整合（Token Misalignment）問題を解決し、投機的デコードを組み合わせることで高速かつ最小侵襲な制約付き生成を実現する「DOMINO」を提案した研究である。

---

## 背景

制約付きデコードの既存手法には共通の問題として「トークン不整合（Token Misalignment）」がある。

### トークン不整合問題の具体例

LLMはBPEなどのサブワードトークナイザを使用するため、正規表現やCFGの制約とトークン境界がずれる。

例えば、日付形式 `\d{4}-\d{2}-\d{2}` を正規表現制約として適用するとき：
- LLMの語彙には `"2024"` という4文字を一つのトークンとして含む。
- 正規表現は最初の4桁のみ許可するが、トークン `"2024-01"` は7文字を一括で生成しようとする。
- 素朴なトークンレベルのマスキングではこの不整合が処理できず、精度が低下する。

従来手法（Outlines等）はこの問題を部分的にしか解決できていなかった。また制約付きデコードのオーバーヘッドは最大2倍以上であり、制約なし生成と比べて速度面で大きなコストを伴っていた。

---

## 手法

### 1. ボキャブラリー整合型サブターミナルツリー（Vocabulary-Aligned Subterminal Tree）

DOMINOの核心は、ターゲット文法の語彙に対してサブターミナルツリーをオフラインで構築することにある。

- **サブターミナル（Subterminal）**: 文法の中で「部分的に文字列を受け入れ得るサブゴール」を指す。
- **ボキャブラリー整合**: 各LLMトークンが文法のどのサブターミナルに整合するかを事前に全計算し、トークン→サブターミナルのマッピングテーブルを生成する。
- **推論時**: 現在の文法状態に基づいて有効なサブターミナルを決定し、対応するトークンを即座に参照する（定数時間）。

### 2. 最小侵襲設計（Non-Invasive）

DOMINOは制約を「完全に一致するステップでのみ適用」する設計により、LLMの自然な生成確率への干渉を最小化する。

- 現在のプレフィックスが制約に完全に整合している場合のみトークンマスクを適用する。
- 整合していない場合は通常の生成を行い、事後バリデーションに委ねる。

### 3. 投機的デコード（Speculative Decoding）との統合

DOMINOは投機的デコードを制約生成に応用する。

$$P(\text{accept token } t | \text{state } s) = \min\left(1, \frac{p_{\text{constrained}}(t|s)}{p_{\text{draft}}(t|s)}\right)$$

- 小型のドラフトモデルでトークンを先読みし、大型のターゲットモデルで検証する。
- 制約チェックをドラフトモデルの段階で行い、無効なトークンのドラフトを早期に棄却することで、ターゲットモデルの計算コストを削減する。

## 結果

### スループット向上（主要結果）

| 設定 | DOMINO スループット | 対ベースライン比 |
| :--- | :---: | :---: |
| JSON制約（小サイズ出力） | 制約なし生成の **2倍以上** | +100% |
| Python制約（中サイズ出力） | 制約なし生成と同等〜1.5倍 | +0〜50% |
| SQL制約 | 制約なし生成と同等以上 | +0〜30% |

### 精度（トークン不整合問題の解消）

| 手法 | JSON Schema 適合率 | 速度オーバーヘッド |
| :--- | :---: | :---: |
| 素朴なマスキング | 低（不整合で失敗多発） | 低 |
| Outlines | 高 | 中（1.2〜1.5倍） |
| **DOMINO** | **高** | **なし（場合によりマイナス）** |

## chokosenlovetiのメモ

### GPUを用いた正規表現高速化との関係

DOMINOが解決する「サブワードトークン境界と正規表現境界の不整合」は、GPU正規表現マッチングにおける「パケット境界をまたがるパターンマッチング」問題と類似の課題構造を持つ。

- **境界整合問題**: LLMのトークン境界とFSM状態境界のずれ → GPUパケット処理のスライス境界をまたぐ正規表現マッチング問題
- **投機的デコード**: LLM推論の高速化技術を制約生成に応用した点が新規性であり、GPUの投機的実行（パイプライン処理）との概念的対応がある
