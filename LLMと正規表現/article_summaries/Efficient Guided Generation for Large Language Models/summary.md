# Efficient Guided Generation for Large Language Models

本論文は、LLMの出力を有限状態機械（FSM）のトラバーサルとして定式化し、正規表現・文脈自由文法への制約を効率的に強制する手法「Outlines」を提案した研究である。

---

## 背景

大規模言語モデル（LLM）はツール呼び出しや構造化データ抽出など、特定のフォーマットに従った出力が求められる用途に広く活用されている。しかしLLMは自由なトークン列を生成するため、出力がJSONスキーマや正規表現に合致することを保証できない。

従来の主なアプローチには次の2つがある。
- **事後フィルタリング**：生成後に構文チェックし、違反なら再生成する。計算コストが高く信頼性に欠ける。
- **プロンプトエンジニアリング**：プロンプトに制約を記述するが、準拠率は不安定で保証できない。

デコード中にトークン選択を制限する「制約付きデコード」が有望だが、従来手法は各ステップで全語彙を走査して有効トークンを特定するため $O(N \times |V|)$ のコストがかかり、推論速度の大幅な低下を招いていた。

---

## 手法

### 1. 生成をFSMトラバーサルとして定式化

正規表現 $r$ に対応するFSM $M = (Q, \Sigma, \delta, q_0, F)$ を構築し、LLMの生成を次のように定義する。

- 生成の各ステップでFSMの現在状態 $q$ を保持する。
- 生成したトークン $t$ をFSMに適用した場合の遷移先 $\delta(q, t)$ が存在するか否かでトークンの有効性を判断する。
- 無効なトークンのロジットに $-\infty$ を代入してマスクする。

### 2. 事前インデックスによる $O(1)$ マスク計算

キーアイデアは語彙に対するFSM状態の事前計算である。

$$\text{Index}[q] = \{t \in V \mid \delta(q, t) \text{ が存在する}\}$$

各FSM状態 $q$ から有効に遷移できるトークン集合を事前に計算しておくことで、推論中の各ステップで $O(1)$ のインデックス参照のみでマスクを生成できる。

### 3. 文脈自由文法（CFG）への対応

CFGに対応するプッシュダウンオートマトン（PDA）にも同様の手法を適用し、JSON・SQL・コードなど再帰的な構造をもつ出力形式にも対応する。

### 4. Outlinesライブラリへの実装

提案手法は`outlines`ライブラリとして実装された。

```python
import outlines

model = outlines.models.transformers("mistralai/Mistral-7B-v0.1")

# 正規表現による制約付き生成
generator = outlines.generate.regex(model, r"\d{3}-\d{4}")
phone_number = generator("電話番号を教えてください：")
```

## 結果

### 手法の特性比較

| アプローチ | 有効性保証 | 計算コスト | モデル依存性 |
| :--- | :---: | :---: | :---: |
| プロンプトのみ | なし | 低 | なし |
| 事後フィルタリング + 再生成 | 確率的 | 高（再生成が必要） | なし |
| 従来の制約付きデコード（全語彙走査） | あり | $O(N \times |V|)$ | あり |
| **Outlines（本手法）** | **あり** | **$O(1)$（推論時）** | **なし（モデル非依存）** |

### 性能・採用状況

- 語彙サイズ $|V|$ に依存しない $O(1)$ の推論時マスク計算を達成。
- モデル非依存のため、Hugging Face上のあらゆるTransformerモデルに適用可能。
- vLLM、SGLang、Hugging Face TGI等の主要推論フレームワークのバックエンドとして採用された。
- 制約付きデコードの事実上の標準ライブラリ（2023〜2025年時点）となり、後続の XGrammar・SynCode・CRANE等の研究の出発点となった。

## chokosenlovetiのメモ

### LLMの正規表現高速化への応用可能性

本論文の提案するFSMベースの制約付きデコードは、GPU上での正規表現マッチング高速化（例：DFA遷移テーブルのGPUメモリ最適化）との接点が存在する。

- **FSMの事前計算**: 語彙に対するFSM状態インデックスの事前計算は、GPUにおけるDFAテーブルの事前ロードと概念的に対応する。
- **マスク計算の並列化**: デコードのバッチ処理において複数シーケンスのFSM状態遷移を同時計算するため、GPUによる並列状態遷移計算（iNFAntやYu (2013)の手法）が直接応用できる可能性がある。
- **状態爆発問題**: LLM語彙（数万〜数十万トークン）に対してFSMのインデックスを展開すると、FSM状態数×語彙サイズ分のメモリが必要であり、GPU上での正規表現マッチングで問題となったDFA状態爆発と同種の課題が生じる。
