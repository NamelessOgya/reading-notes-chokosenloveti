# XGrammar: Flexible and Efficient Structured Generation Engine for Large Language Models

本論文は、LLM向け文法制約付き構造化生成の効率的エンジン「XGrammar」を提案した研究である。語彙をコンテキスト独立／依存に分割し、永続スタックと推論エンジンとのGPU計算オーバーラップにより従来比最大100倍の高速化を達成した。

---

## 背景

LLMの構造化出力（JSON・SQL・コード）への需要が急増する中、文法制約デコードの計算オーバーヘッドが本番環境での採用の障壁となっていた。特に文脈自由文法（CFG）に対応するプッシュダウンオートマトン（PDA）ベースの手法では：

- **前処理コスト**: 文法とトークナイザのインデックス構築に数分〜数十分かかる。
- **推論コスト**: 各トークン生成ステップでPDAのスタック状態に対してマスクを計算するため、GPUの生成計算に対してCPUのマスク計算がボトルネックになる（CPU-GPUの同期待ち）。

これにより、実際のLLMサービングにおいてスループットが大幅に低下するという問題があった。

---

## 手法

### 1. 語彙のコンテキスト独立・依存分割

XGrammarの核心はLLMの語彙を2種類に分類することにある。

- **コンテキスト独立トークン（Context-Independent Tokens）**: どの文法状態からでも有効（例：特定のリテラル）なトークン。これらはオフラインで事前チェックし、文法状態に関係なく常に有効と判断できる。
- **コンテキスト依存トークン（Context-Dependent Tokens）**: 現在のPDAスタック状態に依存して有効性が変わるトークン。実行時インタープリタで動的に評価する。

$$V = V_{\text{indep}} \cup V_{\text{dep}}$$

この分割により、コンテキスト独立トークンは前処理段階でビットマスクを固定でき、推論時の計算を大幅に削減できる。

### 2. 効率的な永続スタック（Persistent Stack）

CFGデコードにはPDAのスタック状態管理が必要だが、従来手法はスタックをコピーするため $O(\text{depth})$ のメモリと時間コストがかかった。

XGrammarはツリー構造の永続スタックを採用し：
- スタックの分岐時に共有ノードを再利用することで $O(1)$ のプッシュ/ポップを実現。
- ロールバック操作（バックトラック）を $O(1)$ で実行可能にする。

### 3. 推論エンジンとのGPU計算オーバーラップ

LLM推論エンジン（vLLM等）との共同設計により：
- GPUがトークン確率を計算している間、CPU（XGrammar）が次のステップのマスクを並行して計算する。
- CPU-GPU同期待ちによるレイテンシが実質ゼロになる。

### 4. オートマトン最適化

CFGを変換したPDAに対して以下の最適化を適用する：
- **インライン化**: 頻繁に呼び出される非終端記号の規則をインライン展開し、スタック操作回数を削減。
- **等価状態統合**: 同一の遷移関係を持つ状態を統合し、オートマトンのノード数を削減。

## 結果

### スループット向上（主要結果）

| 比較対象 | XGrammar スループット比 |
| :--- | :---: |
| 既存制約デコード手法（Outlines等） | **最大100倍高速** |
| 制約なし生成（ベースライン） | ほぼ同等〜若干遅い程度 |

### 前処理時間の削減

| 手法 | JSON Schema 前処理時間 | Python文法 前処理時間 |
| :--- | :---: | :---: |
| 従来手法 | 数分〜十数分 | 十数分〜数十分 |
| **XGrammar** | **数秒以内** | **数秒以内** |

### 採用実績

XGrammarはvLLM・SGLang・TensorRT-LLM・MLC-LLMの構造化出力バックエンドとして採用され、産業界標準エンジンとなった。

## chokosenlovetiのメモ

### GPU正規表現高速化との関係

XGrammarが取り組む「CPU-GPU間の同期ボトルネック」は、GPU上の正規表現マッチングで取り組む「CPU-GPU間のPCIeデータ転送ボトルネック」と同一の問題構造を持つ。

- **ダブルバッファリング**: XGrammarがGPUのトークン計算とCPUのマスク計算をオーバーラップさせる手法は、GPU正規表現研究（Vasiliadis 2009, Yu 2013）のダブルバッファリングと全く同一の設計思想である。
- **語彙分割のアナロジー**: コンテキスト独立/依存トークンへの分割は、GPU DFAマッチングにおける「デフォルト遷移（多くの状態で共通）」と「明示的遷移（状態依存）」の分離（C-DFA）と対応する。
- **GPU上でのXGrammar実現**: 将来的に語彙マスク計算をGPUカーネルで並列実行すれば、CPU-GPU同期ボトルネックをさらに解消できると考えられる。正規表現の並列マッチング技術（NFA並列状態遷移）が活用できる可能性がある。
