# PICARD: Parsing Incrementally for Constrained Auto-Regressive Decoding from Language Models

本論文は、LLMのSQL生成において逐次的なSQLパーサ検証をデコードに組み込む制約付き自己回帰デコード手法「PICARD」を提案した研究である。追加学習なしでSpiderベンチマークのSOTA精度を達成した。

---

## 背景

Text-to-SQL（自然言語からSQLへの変換）はLLMの重要なユースケースであるが、生成されたSQLに文法エラーや意味的な誤り（存在しないカラム名の参照等）が含まれると実行できない。

T5などの事前学習済み言語モデルをFine-tuningする手法が主流だったが：
1. **文法的誤り**: 括弧の対応ミス・予約語の誤用
2. **意味的誤り**: スキーマに存在しないテーブル・カラムの参照

これらの誤りはFine-tuningのみでは完全に除去できず、後処理による修正やクエリの実行不可という問題が残っていた。

---

## 手法

### 1. PICARD（逐次的インクリメンタルパーシング）

PICARDはビームサーチの各ステップでSQLパーサを実行し、生成途中のクエリを即座に検証する。

$$\text{Beam}_k^{(t+1)} = \{(y_{1:t}, y_{t+1}) \mid y_{t+1} \in \text{Valid}(y_{1:t}), \text{score}(y_{1:t+1}) \in \text{Top-K}\}$$

ここで $\text{Valid}(y_{1:t})$ は現在のプレフィックス $y_{1:t}$ から続けて生成可能な文法的・意味的に有効なトークン集合。

### 2. 4段階の制約レベル

PICARDは制約の厳しさを4段階で調整できる：

| レベル | 制約内容 |
| :--- | :--- |
| **PICARD-Off** | 制約なし（通常のビームサーチ） |
| **PICARD-Lexing** | SQLの字句解析のみ（予約語・構文トークンの有効性チェック） |
| **PICARD-IncParsing** | SQLのインクリメンタルパーシング（全構文規則をチェック） |
| **PICARD-IncParsingWithGuards** | インクリメンタルパーシング + スキーマ整合性チェック（カラム・テーブルの存在確認） |

### 3. インクリメンタルパーサの実装

SQLのパーサをインクリメンタル（逐次的）に動作するよう実装することが技術的課題。

- **Earleyパーサ**: 任意のCFGに対してインクリメンタル解析が可能なアルゴリズムを採用。
- **アイテムセットのキャッシュ**: 前ステップのパーサ状態を保持し、次のトークン追加時に差分計算のみを行うことで効率化。
- **スキーマ整合性**: データベーススキーマ（テーブル・カラム定義）をパーサに統合し、実際に存在するエンティティのみを有効とする。

## 結果

### SpiderベンチマークでのEM（Exact Match）精度（主要結果）

| モデル | PICARD | Dev EM | Test EM |
| :--- | :---: | :---: | :---: |
| T5-3B | Off | 57.3% | 55.8% |
| T5-3B | **PICARD-IncParsingWithGuards** | **66.6%** | **65.3%** |
| T5-11B | Off | 62.9% | 61.0% |
| T5-11B | **PICARD-IncParsingWithGuards** | **75.1%** | **75.5%** |
| BRIDGE（従来SOTA） | - | 71.1% | 67.5% |

- PICARD適用により T5-3Bで **+9.3ポイント**、T5-11Bで **+12.2ポイント** の精度向上
- **追加学習なし**でSOTAを達成（T5のFine-tuningのみで実現）

### 制約レベル別精度（T5-3B, Dev セット）

| 制約レベル | Dev EM |
| :--- | :---: |
| Off | 57.3% |
| Lexing | 60.8% |
| IncParsing | 64.1% |
| IncParsingWithGuards | **66.6%** |

制約が厳しくなるほど精度が向上し、スキーマ整合性チェックが最も効果的であることが示された。

## chokosenlovetiのメモ

### GPU正規表現高速化との関係

PICARDはSQLの文法をCFGとして表現し、Earleyパーサを逐次的に実行する。これはGPU上での正規表現マッチングと以下の点で関係する。

- **インクリメンタルパーシングの並列化**: ビームサーチの各ビーム候補を並列にパーサで検証する操作は、GPU上でのNFA並列状態遷移と同一の計算パターンである。各ビーム候補を独立したスレッドに割り当ててGPUで並列検証すれば、PICARDの推論速度を大幅に向上できる可能性がある。
- **DFA事前計算の応用**: SQL文法のDFAをGPUメモリに常駐させ、各トークン追加時の有効性チェックをGPUカーネルで並列実行することで、現在CPUで行っているEarleyパーシングをGPUオフロードできる。
