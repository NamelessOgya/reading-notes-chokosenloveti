# Reflection

- 課題: Markdownで同階層の画像ディレクトリにアクセスする際、誤って `../images/`（一つ上の階層）と指定して画像が表示されないミスがあった。
- 対策: 今後はファイルからの相対位置を必ず確認し、同階層の場合は `./images/` のように正しく記述する。

- 課題: macOS標準の`sips`コマンドでPDFの論文図版を画像化する際、ベクター等の複雑な処理ができず真っ黒や真っ白な画像が生成されてしまう問題が発生した。
- 対策: 今後PDF画像を正しく変換する際は、プレビュー描画に強い `qlmanage -t -s 1200 -o [出力先] [PDFパス]` コマンドを使用して、安定して高画質なPNGサムネイルを生成する。

- 課題: フォーマットの指示でTableのまとめが漏れたり、データを一部抜粋してしまい情報が欠落する問題があった。
- 対策: 今後は論文中のTableを扱う際、**データを一切抜粋せず全行・全列を完全なMarkdownテーブルとして転記**し、FigureだけでなくTableに関する考察も必ずセットでまとめること。

- 課題: スクリプト実行などでPythonが必要になった際、ローカル環境を汚してしまう懸念がある。
- 対策: 今後は提供されている `docker-compose.yml` を利用し、`docker compose run --rm python bash` コマンドを使用してコンテナ内で作業を完結させる。

- 課題: arXiv論文のソースと図版PDFをダウンロードし、手動で画像変換や引用抽出を行うことは、環境依存のミスや情報の欠落が起きやすく非常に非効率である。
- 対策: 今後、新たに論文を調査・要約する際は必ず `./tools/extract_arxiv.sh <arXiv_ID> <タスク名(論文名)>` を実行し、ソースの安全な取得、`qlmanage`経由の画像変換、Docker環境下での各種抽出を一元的に済ませてから要約作成に取り組むこと。

- 課題: `tables_source.md` を盲信したり、`images/` ディレクトリ内の画像を独断で取捨選択（チェリーピッキング）した結果、表の番号が論文原著とズレたり、重要な図が要約から欠落するミスが発生した。
- 対策: 新たに要約を作成する際は、必ず大元の `.tex` ソースにおける `\caption` や `\label`、行頭のコメントアウト（`%`）の有無を直接確認して「原著における真の図表番号」を把握すること。また、生成された画像は一切チェリーピッキングせず、原著と同じ番号（Figure 4, Table 2など）を踏襲してすべてMarkdown要約に組み込むこと。

- 課題: 発表直後の論文の「被引用論文（Cited by）」を検索した際、結果が存在しないにも関わらず、論文内のReference（ベースライン等）を「被引用論文である」と誤認・捏造してリストアップしてしまうミスがあった。
- 対策: 今後被引用論文を探す際は、必ずWeb検索等で「Cited by」の事実を確認すること。見つからなかった場合は無理に結果を捻出せず、「まだ当該論文を引用する後続研究は存在しない」と事実を正確に述べること。その上で、代替となる同時期の関連論文（Concurrent Work）や同著者のフォローアップを提案するように自律的にフォールバックを行うこと。

- 課題: Markdownに埋め込む画像パスを記述する際、過去の抽出結果（PDFからのqlmanageによるPNG変換）の先入観から実際の画像ファイルの拡張子（.jpgなど）を確認せず一律で `.png` と記述してしまい、画像が欠損するミスがあった。
- 対策: 今後はMarkdown内に画像のリンクを記載する際、必ず対象ディレクトリ内の出力結果を確認し、拡張子（.jpg, .png等）や大文字小文字を含めた正確なファイル名を取得してから記述する。先入観で拡張子を決めつけないこと。
